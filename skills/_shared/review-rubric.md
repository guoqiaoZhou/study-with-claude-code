# 评审准则（Review Rubric）

> 本文件供 `plan` 技能在「强制 subagent 评审」阶段使用，定义评审维度、各 level 深度校准、缺口清单格式。`plan` 派发评审子智能体时把相关维度的 prompt 模板交给子智能体，并要求其按本文的 `gaps[]` 结构返回。

---

## 一、为什么要 subagent 评审

主 Claude 生成知识树时容易**自我满足**——列完一遍就觉得齐了。独立子智能体带着「挑刺」视角复审，能稳定地暴露主体看不到的覆盖缺口与深度不足。**这是技能用流程保证质量，不是让用户来判断好坏。** 评审是强制环节，不询问用户是否需要。

---

## 二、三个评审维度

每个维度派一个子智能体，**并行**派发。每个子智能体只读、不改文件，只返回结构化缺口清单。

### 维度 1：覆盖度（coverage）—— 始终评审
> 你是某技术专题的资深面试官。下面是为 `<topic>`（目标级别 `<level>`）生成的知识树大纲与详细知识体系。请对照**大厂后端/对应岗位 `<level>` 的真实面试与实战要求**，找出**缺失的高频主题或节点**：哪些该考的点没出现？哪些子主题明显不完整？只报真实缺口，不要为凑数而列边角冷门点（除非 level 要求）。

### 维度 2：深度（depth）—— 始终评审
> 下面是 `<topic>` 的知识体系。请逐节点判断:它是**停留在罗列/背诵**（只有结论、名词），还是**讲清了原理、设计取舍、边界条件、可举例**？标出**太浅、需要加深**的节点，并说明该补什么（原理推导 / 为什么这样设计 / 与替代方案的取舍 / 失效边界 / 真实场景）。

### 维度 3：书本忠实度（book-fidelity）—— 仅当挂了参考资料时评审
> 本专题挂了参考资料 `<refs>`。下面的大纲是「先独立生成、再用书锚定」得到的。请检查两件事:① 大纲是否**过度受限于书的目录**（书没讲的重要主题被漏掉）？② 书里**确有但大纲漏掉**的重点章节/概念有哪些？提醒:书是锚点不是天花板，超出书的内容只要是该 level 该掌握的就应保留。

---

## 三、各 level 深度校准

| level | 定位 | 节点粒度与深度期望 |
|---|---|---|
| `p6` | 基础/初级 | 覆盖主流概念与常用法；每节点讲清「是什么、怎么用、典型场景」。子主题 3–5 个，叶子节点适中。 |
| `p7` | 进阶/高级（默认） | 在 p6 基础上深入**原理与取舍**；每节点要有「为什么这样设计、与替代方案对比、边界与坑」。覆盖高频面试考点 + 实战调优。密度见下方「深度标准」。 |
| `p8` | 专家 | 源码级、底层实现、性能调优与故障排查、版本演进；包含冷门但有区分度的点；能讲清设计哲学与历史包袱。 |
| `p9` | 架构/资深专家 | 在 p8 上加跨系统权衡、容量与稳定性、方案选型决策链、生产事故复盘级深度。 |

> 评审子智能体应**按目标 level 这一行**来判断「够不够深、够不够全」，不要用更低或更高 level 的标准误判。

### 深度标准（文字定义，学科无关——本插件唯一的密度基准）

凡提到「深度达标 / roadmap 级密度」，一律指**同时满足**以下五条，与具体学科无关:

1. **叶子节点是可单独考核的知识点**，而不是含义过宽、没法单独出题的大词。
2. **每个节点有一个驱动性核心问题**，定调本节点要回答什么。
3. **核心概念是多条实质要点**，且**每一条都不止给结论**，而是讲清:① 背后的原理 / 机制(为什么是这样)，② 与替代方案的设计取舍(为什么选它、代价是什么)，③ 边界条件 / 失效情形 / 易错点。
4. **配该 level 真实会被问到的具体面试考点**，不是泛泛的"了解 X"。
5. **标注深度内容的来源指引**(来自哪类资料/章节，或属于资料未覆盖的补充)。

**不达标的样子**:一个子主题只列三五个光名词;每条核心概念只有一句结论，没有原理、没有取舍、没有边界;考点空泛。

> 这是一套硬标准:评审时逐节点对照这五条,缺哪条就报哪条缺口。

## 四、缺口清单格式（gaps[]）

每个评审子智能体返回一个 JSON 数组，元素结构:

```json
{
  "dimension": "coverage | depth | book-fidelity",
  "node": "缺口所属/建议挂载的节点路径（覆盖度缺口可填建议父节点，如「核心机制」）",
  "severity": "high | medium | low",
  "suggestion": "具体要补什么——补哪个主题/节点，或某节点要加深到什么程度"
}
```

- `severity`:`high` = 该 level 必考/必会却缺失或严重过浅；`medium` = 重要但非致命；`low` = 锦上添花。
- 没有缺口时返回 `[]`。

---

## 五、plan 主体如何消费缺口

1. 合并三个维度的 `gaps[]`，按 `node + suggestion` 去重。
2. 优先处理 `high`、其次 `medium`；`low` 视 level 与篇幅酌情。
3. 据缺口**补全 knowledge-tree.md（新增节点）与 knowledge-system.md（加深内容）**。
4. **循环复审**:补完后可再派一轮评审，直到**连续一轮无新增 high/medium 缺口**（loop-until-dry），**最多 2–3 轮**防止失控。
5. 在最终摘要里说明:评审补了哪些缺口（几个 high/medium）。
